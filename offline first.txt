What "offline-first" actually means for AceICT
Offline-first doesn't mean the server disappears. It means:

"Every action works offline. Going online just synchronises."

The browser becomes a full local database. The server becomes a sync target, not a dependency.

What can be 100% offline
Feature	Offline strategy
Take a test	Store attempt + answers in IndexedDB, submit when online
Question bank	Cached locally on last sync
Practice mode	Runs entirely from cached questions
Student dashboard	Serves last-synced data
View results/history	Reads from local cache
Teacher question editor	Queues saves, syncs on reconnect
Mark essays	Queue marks locally, push when online
Grade boundaries, settings	Cached on first load
What fundamentally requires a connection
Feature	Why
Live quiz	Real-time by design — not fixable
Google Meet	External service
AI question generation	Anthropic API call
Receiving new test assignments	Teacher must push them first
Real-time notifications	Need a server to push from
The architecture
Current flow

Browser → API call → PHP → MySQL → Response
Offline-first flow

Browser → IndexedDB (local) → Render immediately
                ↓ (when online)
         Sync queue → PHP → MySQL
         Pull changes → Update IndexedDB
Three pieces make this work:

1. Service Worker — intercepts every fetch. If offline, serves from cache. If online, fetches and updates cache.

2. IndexedDB — a full relational-capable database inside the browser. Stores questions, tests, attempts, results, student data — everything.

3. Sync Queue — every write (submit attempt, save question, record attendance) is written locally first, then queued. Queue drains when online.

What needs to be built
On the frontend

// Before (always needs network)
const tests = await GET('/tests');

// After (offline-first)
const tests = await db.tests.getAll();           // instant, from IndexedDB
syncInBackground('/tests').then(fresh => {       // update if online
  db.tests.putMany(fresh);
  rerenderIfStillOnPage();
});
A sync manager that runs every time the browser comes online:


self.addEventListener('online', () => {
  drainSubmitQueue();   // push queued test submissions
  drainMarkQueue();     // push queued marks
  pullLatestData();     // pull new tests, assignments, questions
});
On the backend
Almost nothing changes. The PHP API stays as-is. We add:

A GET /sync?since=timestamp endpoint that returns everything changed after a given time
Each table gets an updated_at column (most already have it)
Soft deletes (mark as deleted, don't actually delete, so the client can sync the deletion)
Conflict resolution rules
This is the hard part. When two things edit the same record offline:

Conflict	Rule
Student submits test twice	First submission wins, second rejected
Teacher edits question offline, admin edits same question	Last-write-wins (show warning)
Student syncs old result, server has newer mark	Server wins
Two teachers create question with same text	Both kept, flagged as duplicate
Most conflicts in an assessment system are rare because each user owns their own data — students own their attempts, teachers own their questions.

Progressive Web App (PWA) shell
The app itself gets installed like a native app:


// manifest.json
{
  "name": "AceICT",
  "start_url": "/",
  "display": "standalone",
  "background_sync": true,
  "offline_enabled": true
}
Students on Android tap "Add to Home Screen." It installs. Works without internet. Syncs in the background when school WiFi returns. No app store needed.

When they reconnect:

Honest limitations
First visit always needs internet (to download the app)
JWT tokens expire — need a 30-day offline token strategy
If a student is offline for weeks, they miss new assignments until they sync
IndexedDB has storage limits (~50–500MB depending on device) — enough for a full question bank but not unlimited
Bottom line
Yes — it's entirely feasible. The existing PHP backend barely changes. The frontend gets a sync layer and a local database. The hardest part is conflict resolution, and for an assessment system, the rules are actually simple because each user mostly writes their own data.

Want me to start with Phase 1? — the service worker and app shell caching. That alone makes the app load instantly and work when the network drops, with no changes to any existing features.
